This is a well written document, but I have a few minor comments I would like to discuss:

In 5.1:

  The procedures for encoding the JWS and calculating the signature are
  defined in [RFC7515].  As the JWS Payload information is found in
  other SIP information elements, the JWS Payload is not attached to
  the serialized JWS conveyed in the header field parameter, as
  described in Appendix F of [RFC7515].  The operator identifier and
  the serialized JWS are separated using a colon character.

I read this several times and I am still not sure what the second sentence is trying to say.
If you are saying that you are using detached signature, I think it is better to say so.

In 5.6.2:

  jws            = header ".." signature
  header        = *base64-char
  signature      = *base64-char

If neither "header" nor "signature" can be empty, you should use "1*" when defining them.

In 9:

  A SIP entity MUST use different key values for each parameter value
  that it recognizes and use to trigger actions.

I am not sure what this is trying to say. Are you advising against key reuse between this document and keys used for other purposes?

Upon approval of this document, we understand that there are two registry actions to complete.

First, in the Header Field Parameters and Parameter Values subregistry of the Session Initiation Protocol (SIP) Parameters registry located at:

https://www.iana.org/assignments/sip-parameters/

a single, new Header Field and Parameter Name is to be registered as follows:

Header Field: Via
Parameter Name: received-realm
Predefined Values: No
Reference: [ RFC-to-be ]

Second, in the JSON Web Token Claims subregistry of the JSON Web Token (JWT) registry located at:

https://www.iana.org/assignments/jwt/

five new JSON Web Token Claims will be registered as follows:

Claim Name: sip_from_tag
Claim Description: SIP From tag header field parameter value
Change Controller: [ IESG ]
Reference: [ RFC-to-be ], [ RFC 3261 ]

Claim Name: sip_date
Claim Description: SIP Date header field value
Change Controller: [ IESG ]

Claim Name: sip_callid

Abstract


  This specification defines a new Session Initiation Protocol (SIP)
  Via header field parameter, "received-realm", which allows a SIP
  entity acting as an entry point to a transit network to indicate from
  which adjacent upstream network a SIP request is received, using a
  network realm value associated with the adjacent network.




The file can be obtained via
https://datatracker.ietf.org/doc/draft-holmberg-dispatch-received-realm/

- Section 2:

Here is my AD Evaluation of draft-holmberg-dispatch-received-realm-06. I'd like to discuss the substantive comments prior to starting IETF LC.

Substantive Comments:

- Section 2:

It’s reasonable that this draft not specify a standard mechanism. But can it offer an example or two of real-world potential methods? (I ask this in anticipation of last call and IESG questions about whether this can be reasonably be kept out of scope.)

- 5.3: Note about operators agreeing on algorithms:
Are the algorithms labels assumed to be those defined in RFC 7518 (or in the registry defined there)?

- 5.6.3: The abnf for "jws" seems odd. It sort of shortcuts the JWS syntax, I assume due to the assumption of a detached payload. It would be worth mentioning that here. Did people consider just saying "jws with detached payload as defined in 7515 appendix F, rather than partially expanding it here?

- 6.2, 2nd paragraph: Is the point of this paragraph to say that only HMAC algorithms are allowed? If so, it would be helpful to say that earlier when mentioning agreeing on algorithms. ( this implies a similar question about the secret key requirement vs public/private keys)


Editorial Comments and Nits:
- 4: Is the IOI definition still relevant, since the draft declares it can't be used?

- 5.1, 2nd paragraph:
7515 appendix F uses the term “detached payload”. It would be helpful to use the same term here.

- 6.2, 2nd paragraph:
It's a bit confusing to say "add the HMAC to the parameter", when there is no field called HMAC. It would help to say this goes in the signature field (I assume?).
