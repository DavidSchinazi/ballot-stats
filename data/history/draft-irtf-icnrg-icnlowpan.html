(Via drafts-eval@iana.org): IESG/Authors/ISE:

The IANA Functions Operator has completed its review of draft-irtf-icnrg-icnlowpan-10. If any part of this review is inaccurate, please let us know.

We understand that when this document is sent to us for processing, we will add the following entries to the Dispatch Type Field registry at

https://www.iana.org/assignments/_6lowpan-parameters

The designated experts have reviewed these registrations and recommended that page 14 be assigned if that value is still available when the document is sent to us for processing:

+-------------+------+-------------------------------------------+
| Bit Pattern | Page | Header Type                              |
+-------------+------+-------------------------------------------+
|  00 000000  | TBD1 | Uncompressed NDN Interest messages        |
|  00 100000  | TBD1 | Uncompressed NDN Data messages            |
|  01 000000  | TBD1 | Uncompressed CCNx Interest messages      |
|  01 100000  | TBD1 | Uncompressed CCNx Content Object messages |
|  10 0xxxxx  | TBD1 | Compressed NDN Interest messages          |
|  10 1xxxxx  | TBD1 | Compressed NDN Data messages              |
|  11 0xxxxx  | TBD1 | Compressed CCNx Interest messages        |
|  11 1xxxxx  | TBD1 | Compressed CCNx Content Object messages  |
+-------------+------+-------------------------------------------+

Thank you,

Amanda Baber
Lead IANA Services Specialist

[Ballot comment]
This is a good basis for running experiments with ICN in constrained node network environments.
There are a few nits left, at least one of which needs to be addressed before moving this forward to the next process step.

The "C" flag is rather confusing for the reader (set to 0 for compression); could this be an "U" (uncompressed) flag?
Clearly, the erroneous Table 1 reflects the need for such a change.

RFC 8930 and related have been published; this puts the present specification on somewhat firmer ground.
